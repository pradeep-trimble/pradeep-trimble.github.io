⚡ Core Concepts at a Glance

🏛 MVVM + Clean Architecture

UI → ViewModel → Repository → DataSource. Each layer is interface-backed for testability. Koin provides dependency injection throughout.

📡 Offline-First Sync

On first login, all config & work data is downloaded and stored in local SQLite DBs. BaseSyncManager orchestrates pull/push for each data type.

🔄 Data Flow

Remote API → Sync → SQLDelight → Repository → ViewModel → Compose UI (StateFlow). All reactive via Kotlin Flows.

🗺 Navigation

PreCompose NavHost in MainView.kt. Route-based with string URLs (e.g. /login, /module). Events via NavigationListener flow.

💉 Koin DI

dataModule() provides all repos & sync. commonModule() includes all ViewModels. User-scoped databases via UserScope.

🔗 Integrations

Unity backend, Cityworks work management, eBuilder project management, ArcGIS/ESRI maps, Trimble Connect 3D viewer.

🚀 Platform Entry Points

Android (kmp-android)

The kmp-android module is the Android application wrapper. It initialises Koin modules (dataModule(), commonModule()) and calls MainView() inside a Compose Activity.

💡Build: ./gradlew :kmp-android:assembleAutomationDebug

iOS (kmp-ios)

Xcode project at kmp-ios/ios.xcodeproj. The KMP framework is embedded via Gradle task embedAndSignAppleFrameworkForXcode. MainView() is called from Swift code.

💡Build: ./gradlew :kmp-compose:embedAndSignAppleFrameworkForXcode

Desktop (kmp-desktop)

Java/Desktop variant. Packages into a JAR. Useful for development and testing without a mobile device.

💡Build: ./gradlew :kmp-desktop:jar

Testing

Unit tests live in kmp-core and kmp-data. UI Automation tests are in the automation/ directory using Appium & UI Automator.

⚠️Run: ./gradlew :kmp-core:allTests desktopTest

Interface-Backed Design

Every significant class implements an interface. This enables testing and platform-specific implementations.

// Pattern everywhere in the codebase
interface DomainItemRepositoryInterface {
    suspend fun getDomainItem(id: Long): DomainItem?
    fun observeDomainItems(): Flow<List<DomainItem>>
}

class DomainItemRepository(
    private val workDb: WorkDatabase,
    // ...other deps injected by Koin
) : DomainItemRepositoryInterface { ... }

// In DI module:
single<DomainItemRepositoryInterface> {
    DomainItemRepository(get(), get(), ...)
}

ViewModel Pattern

ViewModels use viewModelScope from PreCompose and expose UI state via StateFlow. 

class DomainItemViewModel(
    private val domainItemRepository: DomainItemRepositoryInterface,
    private val configCache: ConfigCacheInterface,
    // ...many more deps
) : ViewModel() {
    private val _uiState = MutableStateFlow(UiState())
    val uiState = _uiState.asStateFlow()

    init {
        viewModelScope.launch {
            domainItemRepository
                .observeDomainItem(id)
                .collect { _uiState.update { ... } }
        }
    }
}

Login Flow – Step by Step

1. User enters credentials on LoginScreen
LoginView renders UI state from LoginViewModel.uiState. User taps "Login" → emits LoginEvent.
2. LoginViewModel calls LoginCommand
LoginCommand determines the login strategy (Unity vs Cityworks vs eBuilder) based on organisation SSO config.
3. LoginManager authenticates
UnityLoginManager, CwLoginManager, or EbLoginManager calls the appropriate API, obtains OAuth tokens, and stores them.
4. Koin UserScope is created
Scoped databases (config.db, work.db, connect.db) are initialised in the user's directory.
5. BaseSyncManager starts initial sync
Downloads config data (form definitions, attribute definitions, domain item types, map layers) and work data (domain items) from the APIs.
6. Navigate to AppChooserScreen or ModuleScreen
NavigationListener.navigationFlow emits a NavigateEvent. MainView's NavHost handles the navigation.

Domain Item CRUD Flow

A "Domain Item" is the central data entity — it represents a Work Order, Inspection, Service Request, Case, or SmartForm record.

Read (viewing an item)

Navigate to DomainItemScreen
Pass internalId or isCreate=true via route query params.
Koin creates DomainItemViewModel
Factory injection with params. ViewModel loads item from DomainItemRepository.
Repository queries SQLDelight work.db
Returns domain item with all attribute values from DomainItem table.
UI renders via AttributesEditor
Each field renders the appropriate TypeView (TypeViewString, TypeViewDate, etc.).

Save / Submit

User modifies field value
TypeView emits change → ViewModel updates state.
User taps "Save" or "Submit"
ViewModel calls DomainItemRepository.saveDomainItem().
Saved to local work.db
Marked as a local draft. Immediately available offline.
Push sync to remote
SubmitInstanceOperation → appropriate Push (WorkOrderPush, InspectionPush, etc.) → remote API.

Navigation Patterns

Global Navigation Events

For navigation triggered from non-Compose code (e.g., after login), use the global NavigationListener flow:

// From any class (e.g., LoginManager)
NavigationListener.navigate(HomeScreen.route)

// In MainView.kt — listening:
NavigationListener.navigationFlow.collect {
    navigator.navigate(it.route)
}

Result-Based Navigation

Some screens return data to callers (e.g., picking a geometry on a map):

// Caller navigates and waits
val result = navigator.navigateForResult(route)

// Destination pops with result
navigator.goBackWith(geometryResult)

💉 Dependency Injection (Koin)

💡The project uses Koin for dependency injection. There are two main module functions: dataModule() (in kmp-data) and commonModule() (in kmp-compose). Platform-specific code is handled via platformCoreModule.

dataModule()

Defined in kmp-data/src/commonMain/.../di/DataModule.kt

Provides singletons for:

  • All Repositories (ConfigData, WorkData, DomainItem, Attachment…)
  • All Sync managers (BaseSyncManager, WorkOrderSync…)
  • Login Managers (UnityLoginManager, CwLoginManager…)
  • API Services (UnityService, CityworksService…)
  • Location (UnityLocationManager, TMMClient)
  • Global & Connect databases

And scoped (UserScope) for:

  • ConfigDatabase (config.db per user)
  • WorkDatabase (work.db per user)
  • ConnectDatabase (connect.db per user)
  • SmartForms services

commonModule()

Defined in kmp-compose/src/commonMain/.../di/CommonModule.kt

Includes dataModule() and adds:

  • All ViewModels (factory — new instance each time)
  • Commands (LoginCommand, CancelSyncCommand…) — single
  • Delegates (AlertDialogDelegate, BarcodeDelegate…)
  • Shared helpers (GISFeatureHelper, SearchManager…)
  • Script engine (LazyScriptManager, PlatformScriptEngine)
  • UserPreferences (PrefUtils global/user)

How ViewModels Get Their Dependencies

In Compose screens, use koinViewModel() from PreCompose-Koin. For VMs needing constructor params, use parametersOf(): 

// Simple ViewModel - no params
val settingsVM = koinViewModel(SettingsViewModel::class)

// ViewModel with runtime params (from nav route)
val domainItemVM = koinViewModel(DomainItemViewModel::class) {
    parametersOf(isCreate, internalId, dataString, parentId)
}

// In commonModule.kt, factory receives params:
factory { params ->
    DomainItemViewModel(
        isCreate = params[0],
        domainItemInternalId = params[1],
        // repos injected automatically with get()
        domainItemRepository = get(),
        configCache = get(),
        ...
    )
}

SQLDelight WAL Mode & Migrations

All databases are opened with Write-Ahead Logging (WAL) for better concurrent read performance. Schema migrations are auto-applied by comparing the PRAGMA user_version against the schema version.

⚠️ Downgrade not supported. If the on-device database version is higher than the code's schema version, a warning is logged. Verify migrations in kmp-data/src/commonMain/sqldelight/*/migrations/.

For config changes during sync, the app writes to config-temp.db first and then swaps it atomically to avoid serving half-synced data to the UI.

🔍 How To — Common Tasks

📺 Where do I add a new screen? ▼
  1. Create a new Kotlin file in kmp-compose/src/commonMain/kotlin/.../ui/screens/MyNewScreen.kt
  2. Define an object MyNewScreen with a const val route = "/myNewRoute"
  3. Add a @Composable fun Content(...) method with your UI
  4. Register the route in MainView.kt inside the NavHost block: scene(route = MyNewScreen.route) { MyNewScreen.Content(...) }
  5. Navigate to it from other screens using onNavigate(MyNewScreen.route)
🧠 Where do I add a new ViewModel? ▼
  1. Create MyNewViewModel.kt in the appropriate UI content folder
  2. Extend ViewModel() from PreCompose
  3. Register it in CommonModule.kt as a factory { MyNewViewModel(get(), get()) }
  4. In your screen, access it with koinViewModel(MyNewViewModel::class)
  5. If it needs constructor params, use factory { params -> MyNewViewModel(params[0], get()) } and pass them via parametersOf()
📦 Where do I add a new Repository? ▼
  1. Create MyRepository.kt and MyRepositoryInterface.kt in kmp-data/src/commonMain/kotlin/.../data/repository/
  2. Define the interface with your methods
  3. Implement the interface, taking needed database/service/repo deps via constructor
  4. Register in DataModule.kt: single<MyRepositoryInterface> { MyRepository(get(), get()) }
  5. Inject it into your ViewModel via Koin get()
🗄 Where do I add a new database table? ▼
  1. Create a new .sq file in the appropriate SQLDelight folder under kmp-data/src/commonMain/sqldelight/com/trimble/
  2. Define your CREATE TABLE statement and queries in the .sq file
  3. SQLDelight will generate type-safe Kotlin code from the SQL
  4. Add a migration .sqm file if the table is added to an existing database
  5. Bump the database version in the SQLDelight configuration
  6. Access the generated queries via the database instance (e.g., workDatabase.myTableQueries.getAll())
📡 Where do I add a new API endpoint? ▼
  1. Find the appropriate API interface file (e.g., kmp-cityworks-api, kmp-ebuilder-api, or kmp-data/.../unity/api/)
  2. Add a new suspend fun annotated with the Ktorfit HTTP annotation (@GET, @POST, etc.)
  3. Add the response model data class if needed
  4. Call the new method from the appropriate Repository or Sync class
🔍 How do I find which ViewModel powers a screen? ▼

Two approaches:

  1. By screen name: Open the Screen file (e.g., DomainItemScreen.kt). Look for koinViewModel(SomeViewModel::class) or check the function parameters.
  2. By ViewModel name: Search for the ViewModel class. The CommonModule.kt shows all VMs and their dependencies.
  3. Via DI: All ViewModels are registered in CommonModule.kt as factory { } or single { }. Reference this file as a map.
🔄 How does the offline-first sync work for a new domain item type? ▼

For each new work item type (e.g., a new Cityworks integration), you need:

  1. A Pull sync class (e.g., NewTypeSync) extending the sync pattern — fetches from API and stores in SQLDelight
  2. A Push class (e.g., NewTypePush) — reads local drafts and POSTs to remote
  3. SQLDelight .sq file with the table definition
  4. Register both in DataModule.kt as singletons
  5. Add to BaseSyncManager's list of sync operations
  6. A Repository to expose data to ViewModels

⚙ Workflow / Action System

💡 What is an Action? An action is a configured operation a user can perform on a domain item — for example "Start Work", "Complete Inspection", "Reject", "Add Related Item". Actions are defined server-side, downloaded into config.db, and executed at runtime by ExecuteDomainItemActionCommand.

📚 Core Concepts

Concept Lives In What It Does

🔁 Action Execution State Machine

Every action runs through this state machine inside ExecuteDomainItemActionCommand.execute(). The state machine is a Flow — it emits ViewState updates so the UI can react.

→

⚠️ Other terminal states: ABORTED (cannot run), REQUIRES_INTERNET (action needs network — see Network), CANCELLED (user cancelled). The loop has a hard cap of 100 iterations as a safety net against bugs that would otherwise spin forever.

⚠️ Common Gotchas

  • BindEffect required on Android: Call BindEffect(controller) at the top of your composable tree, otherwise the controller has no Fragment to use and requests will fail silently.
  • iOS Info.plist is mandatory: Each permission requires the matching usage description (NSCameraUsageDescription, NSLocationWhenInUseUsageDescription, etc.) — without it iOS instantly denies.
  • DeniedAlways flow: On iOS users cannot re-grant inside the app. Catch DeniedAlwaysException and call controller.openAppSettings().
  • Bluetooth on Android 12+: You need BLUETOOTH_SCAN + BLUETOOTH_CONNECT, not legacy BLUETOOTH.

🪞 expect / actual — Platform-Specific Code

💡 expect/actual is Kotlin Multiplatform's mechanism for declaring an API once in commonMain and providing per-platform implementations in androidMain, iosMain, desktopMain. Think of it as a compile-time interface that every target must satisfy.

🧱 The Pattern

commonMain — declare

expect class ConnectivityUtil {
    fun start()
    fun stop()
    val isConnected: StateFlow<Boolean>
}

androidMain — implement

actual class ConnectivityUtil(
    private val context: Context
) {
    actual fun start() { /* ConnectivityManager */ }
    actual fun stop()  { /* unregister */ }
    actual val isConnected: StateFlow<Boolean> = ...
}

You can expect classes, interfaces, functions, properties, top-level vals, and even Koin Module instances. Every target you compile for must provide a matching actual.

📋 Major expect/actual surfaces in this project

Common declaration Why it must be platform-specific
platformCoreModule (Koin Module) Each platform binds different SQLite drivers, file APIs, location managers, JS engines
ConnectivityUtil Android = ConnectivityManager, iOS = NWPathMonitor, JVM = polling
PermissionsController Android = ActivityResult, iOS = per-permission delegates
PlatformScriptEngine Android = J2V8, iOS = JavaScriptCore, Desktop = GraalJS
DriverFactory (SQLDelight) Android = AndroidSqliteDriver, iOS = NativeSqliteDriver, JVM = JdbcSqliteDriver
UnityLocationManager internals Wraps platform GPS APIs
FileUtils / file paths Different sandboxing rules per OS
BindEffect (permissions) Android needs Fragment binding; iOS is a no-op
SharedRes (moko-resources) Generated per-platform with native string lookups

🗺 Source-Set Layout

kmp-data/src/
├── commonMain/  ← shared logic, expect declarations
├── androidMain/ ← actual for Android
├── iosMain/     ← actual for iOS (arm64 + simulator)
├── desktopMain/ ← actual for JVM / Java desktop
├── macosArm64Main/ + macosX64Main/ ← actual for macOS

Hierarchy: commonMain is shared by all; iosMain is shared by all iOS targets; nativeMain (when present) groups iOS + macOS.

🛠 How to Add a New expect/actual API

  1. In commonMain, declare expect class Foo { fun bar(): Result }.
  2. In every active target source set, add actual class Foo { actual fun bar(): Result = ... }. The compiler fails fast if any target is missing.
  3. If you depend on third-party platform APIs, add the dependency in the platform-specific source set inside build.gradle.kts.
  4. For DI, prefer providing the platform binding inside platformCoreModule rather than building a separate Koin module per platform.

⚠️ Pitfalls

  • Don't put platform-specific imports in commonMain — the file won't compile for other targets.
  • expect declarations cannot have a body. Put any shared logic into a sealed parent class or a commonMain helper that the actuals delegate to.
  • Android actual classes often need a Context — get it from Koin (get<Context>()) inside platformCoreModule, never via a top-level lateinit var.
  • Adding a new target (e.g. wasmJs) means adding actuals for every existing expect — plan ahead.

⚡ Why It's Critical for Form Rendering

An AttributesEditor with 50 fields is rendered every recomposition. Without caching, that's 50+ SQL reads per frame. With ConfigCache:

  • Attribute definitions are loaded once per form definition.
  • Domain values (drop-down choices) are loaded once per domain name.
  • Display groups, card configs, filter configs are memoised by id.
  • Cityworks-specific config (work orders, inspections, service requests) is preloaded into dedicated caches like cityworksDomainItemTypeSourceCache.

🛠 Reading From the Cache

class AttributesEditorPresenter(
    private val configCache: ConfigCacheInterface
) {
    fun loadForm(formDefId: String): List<AttributeDefinitionData> =
        configCache.formDefinitionAttributeDefinitionsCache[formDefId]
            ?.values
            ?.toList() ?: emptyList()
}

Calls are synchronous. The first call may block while loading from SQLite — call from a background dispatcher (Dispatchers.IO) when on the hot path.

🧹 Invalidation

  • Sync managers call configCache.invalidate*() after persisting fresh config rows.
  • Logout disposes the UserScope, which drops the ConfigCache instance entirely (along with config.db's connection).
  • There is a stub implementation (ConfigCacheStub) used in tests where we want a deterministic, non-loading cache.

📡 Connectivity & Network

💡 This is an offline-first app. Network is a privilege, not a guarantee. Two systems power this: ConnectivityUtil (am I online right now?) and Ktorfit (declarative HTTP that talks to Unity / Cityworks / eBuilder / ArcGIS / Trimble Connect).

🌐 ConnectivityUtil — The Online/Offline Sensor

expect class ConnectivityUtil {
    fun start()
    fun stop()
    val isConnected: StateFlow<Boolean>
}
object InternetRequiredException : Exception("No internet connection available")
  • Android: wraps ConnectivityManager + NetworkCallback.
  • iOS: wraps NWPathMonitor.
  • Desktop: polls via InetAddress.isReachable().

Use the StateFlow in a ViewModel: connectivity.isConnected.collect { online -> ... }. Commands that require internet (e.g. login, on-demand fetch) throw InternetRequiredException when offline so callers can show a snackbar.

🛰 Ktorfit — Declarative HTTP

The app wraps Ktorfit over Ktor. APIs are interfaces with annotations and Ktorfit generates the client at compile time.

interface UnityApi {
    @GET("workorders/{id}")
    suspend fun getWorkOrder(@Path("id") id: String): WorkOrderJson

    @POST("workorders")
    suspend fun createWorkOrder(@Body body: WorkOrderJson): WorkOrderJson
}
  • Each external system has its own service interface + Koin single binding (e.g. UnityApi, CwApi, EBuilderApi, ArcGISApi, TrimbleConnectApi).
  • Auth tokens are added by Ktor plugins — see OAuthAuthenticator + kmp-oauth.
  • JSON is parsed by kotlinx.serialization with ignoreUnknownKeys = true.

The most complex TypeView. Renders a list of materials, each with its own nested AttributesEditor using CompositionLocalProvider(ParentAttributes provides …). Demonstrates:

  • Nested ViewModels (StoreroomAttributesViewModel, ChildAttributesViewModel) loaded with koinViewModel per item
  • Bottom-sheet editing with ResponsiveBottomSheetDialog
  • Per-row sync status badges (Requested / Pending / Fulfilled / Rejected)
  • JSON value parsing/serialisation via parseStockFromValuesMap

TypeViewSignatureComposable

Pure-Compose canvas drawing. Captures touch input into a Path, renders it, and emits a base64 PNG on onValueChanged. Good template for building offline-first capture widgets.

🛠 Adding a New TypeView

  1. Add the new value to FieldType enum (kmp-data).
  2. Create TypeViewYourFieldComposable.kt under kmp-compose/.../ui/common/typeviews/yourfield/.
  3. Wire it into the when in TypeViewComposableFactory.CreateTypeView.
  4. If it needs its own DI / state, register a factory ViewModel in commonModule().
  5. Add validation logic in the matching AttributeDefinitionData path so onValueChanged emits errors correctly.

📋 New Engineer — Onboarding Checklist

Work through this top-to-bottom on day 1. Each item is a small, concrete step — tick it before moving on.

🛠 1. Local environment

  • ☐ Install JDK 17 (the project pins this).
  • ☐ Install Android Studio (latest stable) + Android SDK 34.
  • ☐ Install Xcode (latest stable) + Command Line Tools (xcode-select --install).
  • ☐ Install CocoaPods (brew install cocoapods) — needed for the iOS build.
  • ☐ Clone the repo and run ./gradlew help to download the wrapper.
  • ☐ Open the project in IntelliJ IDEA (preferred) or Android Studio.

🚀 2. First successful build

  • ☐ Android: ./gradlew :kmp-android:installDebug on a connected device/emulator.
  • ☐ iOS: open kmp-ios/ios.xcodeproj in Xcode, pick a simulator, hit Run.
  • ☐ Desktop (smoke test): ./gradlew :kmp-desktop:run.
  • ☐ Run the unit-test sweep once: ./gradlew check.

🔑 3. Get an account & log in

  • ☐ Get test Unity, Cityworks, and eBuilder credentials from your team lead.
  • ☐ Log in on Android — confirm you can pull a module / work order.
  • ☐ Open Settings → flip Dark Mode and Use Metric — confirm the UI updates.

📚 4. Read these (in this order)

  1. ☐ Overview — what the app does and the tech stack.
  2. ☐ Modules — the gradle module map.
  3. ☐ Architecture — Clean Architecture / MVVM layering.
  4. ☐ DI & Koin — how to find and add dependencies.
  5. ☐ Database — config.db / work.db / connect.db / global.db.
  6. ☐ Sync — pull/push pipelines.
  7. ☐ UI Rendering + TypeViews.
  8. ☐ How To — quick recipes.

🔍 5. Trace one request end-to-end

Pick "open a work order" and follow the call:

  1. ☐ ModuleScreen → tap a row → NavHost route to DomainItemScreen.
  2. ☐ DomainItemViewModel → DomainItemRepository.findDomainItem().
  3. ☐ Repository reads from WorkDatabase; metadata comes from ConfigCache.
  4. ☐ AttributesEditor → TypeViewComposableFactory → individual TypeViews render.
  5. ☐ Edit a field → onValueChanged → ViewModel updates StateFlow → recomposition.

🧪 6. Make a tiny first change

  • ☐ Pick a small Jira (or a "good-first-issue").
  • ☐ Branch off main: git checkout -b TUF-XXXX-short-description.
  • ☐ Run ./gradlew detekt + ktlintCheck before pushing.
  • ☐ Open a draft PR; ask for review on Slack.

📞 7. Know who to ask

  • ☐ Bookmark the #unity-mobile Slack channel.
  • ☐ Skim the stories/ folder — it contains the in-flight epics with implementation notes.
  • ☐ Find the on-call rotation page; know who to ping after-hours.
🎯 Goal: by the end of week 1 you should be able to add a new field to a form, wire it through the repository, see it in the UI, and submit a PR. If you're stuck for >2 hours on environment setup, ask — it's almost always a known-issue.

📖 Glossary

Domain words you'll hear every day on this team. Tap a row's term in search to jump to where it's used.

🏗 Architecture & Tech

Term Meaning
KMP Kotlin Multiplatform — share Kotlin code across Android, iOS, Desktop.
Compose Multiplatform JetBrains' port of Jetpack Compose to non-Android targets. The whole UI is built on it.
expect / actual KMP keywords for declaring a platform-specific API in commonMain.
SQLDelight Generates type-safe Kotlin from .sq SQLite files.
Ktorfit Retrofit-style declarative HTTP client over Ktor.
Koin Dependency injection framework. single = singleton, factory = new each call, scoped = lifecycle-bound.
PreCompose Navigation library for Compose Multiplatform (NavHost, navigate()).
moko-resources Generates SharedRes for cross-platform string / image lookup.
Modus Trimble's design system — colors, typography, components.

🧠 Domain Model

Term Meaning
Domain Item The polymorphic core entity — work order, inspection, service request, case, task… All persist in WorkDatabase.
Domain Item Type Configuration row (in config.db) that defines what a Domain Item is — its forms, classification, statuses, actions.
Classification A grouping of Domain Item Types (e.g. "Maintenance" vs "Inspection").
FormDefinition Server-defined description of fields and pages for a Domain Item.
AttributeDefinition One field's config: data type, validation, label, scripts.
AttributeUiState Runtime state for a field — current value, errors, warnings, visibility, isEnabled.
DomainValue One row in a coded-value drop-down (id + label).
Action Configured workflow operation on a domain item — see Workflow.
Module The user-selectable set of Domain Item Types (e.g. "Sewer Maintenance"). Maps roughly to a "project" in other systems.
UserScope Koin scope tied to the logged-in user; owns config.db, work.db, connect.db.

🔄 Data Flow

Term Meaning
Pull / Push Pull = server → device. Push = local edits → server. Both run inside BaseSyncManager.
Sync Status Per-row flag in WorkDatabase: Synced / Unsynced / Conflict / Error.
SyncProgress Global SharedFlow<SyncEvent> the UI listens to for progress dialogs.
ConfigCache In-memory cache of form metadata. See ConfigCache.
OAuthProgress Flow that emits token-refresh / logout events.
NavigationListener Global flow used outside Compose to request navigation (e.g. from background sync errors).
